在真实的知识库管理平台中，文档资料往往呈现极其明显的“两极分化”：

+ **短文本**：比如一条 150 字的“春节红包 Push 补贴规则”、一个 FAQ 问答对、或者一条系统错误代码解释。
+ **长文本**：比如长达 50 页的“2025 年 Q4 商业化大促复盘分析报告”。

**如果用“一刀切”的固定分块策略**，短文本会被生硬截断，或者因为长度不够填满一个 Chunk 而带入大量空白噪音；长文本则会面临上下文割裂问题。

因此，**可以采用“全文向量（针对短文档） + 父子块检索（针对长文档）”的双轨制设计。**

### 一、 核心架构思想：分轨入库，统一召回
这套设计的精髓在于：**在离线建库阶段“分道扬镳”，在在线检索阶段“殊途同归”。**

对于底层的 Milvus 或 Elasticsearch 来说，它并不关心存进来的是一篇完整的短文档，还是长文档切出来的一个子句子。对它而言，那只是一个 `Vector` 和一段 `Text`。 真正的魔法，藏在我们为每一条记录精心设计的 **Metadata（元数据）** 里。

#### 1. 统一的 Metadata 设计规范
为了让检索时的后端代码能够分辨出“谁是谁”，我们必须在存入向量库时，强制加上类似这样的 Metadata 标识：

+ `record_type` (记录类型)：枚举值，分为 `FULL_DOC` (全文短文档) 和 `CHILD_CHUNK` (长文档的子块)。
+ `parent_id` (父块关联 ID)：如果是短文档，此字段为空；如果是子块，此字段记录它对应的父块 ID（用于去 KV 数据库回表查询）。
+ `doc_id` (全局文档溯源 ID)：用于前端展示出处。

### 二、 离线数据加工流水线 (双轨入库)
在数据清洗微服务中，我们需要加入一个**路由判断（Length-based Routing）**。

**Java 伪代码实现：**

```java
public void processAndIngestDocument(String docId, String text) {
    // 设定一个合理的长度阈值（比如 800 字符，通常是一个 Embedding 模型的最佳处理长度）
    int THRESHOLD = 800;

    if (text.length() <= THRESHOLD) {
        // ---------------------------------
        // 策略 A：短文本，直接全文向量化
        // ---------------------------------
        Metadata meta = new Metadata();
        meta.add("doc_id", docId);
        meta.add("record_type", "FULL_DOC"); // 标识这是完整文档

        TextSegment segment = TextSegment.from(text, meta);
        vectorStore.add(embeddingModel.embed(text).content(), segment);

    } else {
        // ---------------------------------
        // 策略 B：长文本，走父子块切分逻辑
        // ---------------------------------
        // 1. 切分出较大的父块 (存入 KV 库，如 MySQL 或 Redis)
        List<String> parentChunks = splitIntoLargeChunks(text);

        for (String parentText : parentChunks) {
            String parentId = UUID.randomUUID().toString();
            kvDatabase.save(parentId, parentText); // 存入文档型数据库

            // 2. 切分出较小的子块 (存入向量库)
            List<String> childChunks = splitIntoSentences(parentText);
            for (String childText : childChunks) {
                Metadata meta = new Metadata();
                meta.add("doc_id", docId);
                meta.add("record_type", "CHILD_CHUNK"); // 标识这是子块
                meta.add("parent_id", parentId);        // 挂载父块 ID

                TextSegment childSegment = TextSegment.from(childText, meta);
                vectorStore.add(embeddingModel.embed(childText).content(), childSegment);
            }
        }
    }
}
```

### 三、 在线检索流水线 (统一解析)
当在后台提问时，后端的检索器去向量库里召回了 Top 5 的记录。这 5 条记录里，可能混杂着 2 条短文档的全文，以及 3 条长文档的子句。

此时，我们需要在代码里进行“多态解析与父块回表”，并将最终的真实上下文组装给大模型。

**Java 后端解析逻辑：**

```java
public String retrieveUnifiedContext(String query) {
Embedding queryVector = embeddingModel.embed(query).content();

// 1. 从向量库统一召回 Top 5 记录（底层并不区分长短）
List<EmbeddingMatch<TextSegment>> matches = vectorStore.findRelevant(queryVector, 5);

// 用于存放最终去重后的完整上下文
Set<String> finalContexts = new LinkedHashSet<>(); 

// 2. 遍历召回结果，根据 Metadata 决定如何处理
for (EmbeddingMatch<TextSegment> match : matches) {
    Metadata meta = match.embedded().metadata();
    String recordType = meta.getString("record_type");

    if ("FULL_DOC".equals(recordType)) {
        // 如果命中短文档：直接把原文加入上下文
        finalContexts.add(match.embedded().text());
        log.info("命中短文档全文，DocID: {}", meta.getString("doc_id"));

    } else if ("CHILD_CHUNK".equals(recordType)) {
        // 如果命中长文档的子块：不能用碎片，必须去 KV 库查出它的父块
        String parentId = meta.getString("parent_id");
        String fullParentText = kvDatabase.getById(parentId);

        if (fullParentText != null) {
            finalContexts.add(fullParentText);
            log.info("命中长文档子块，已回表补全父级上下文，ParentID: {}", parentId);
        }
    }
}

// 3. 拼接并返回最终发送给 LLM 的 Context
return String.join("\n\n---\n\n", finalContexts);
}
```

### 四、 这种架构设计的巨大优势
1. **极高的检索精度**：短文档保留了 100% 的语义浓度；长文档通过子句匹配，像探照灯一样精准锁定了长篇复盘报告中的核心论点。
2. **完美的上下文完整性**：无论命中哪种类型，最后喂给大模型的材料，要么是一份完整的短规则，要么是一大段连贯的长报告章节，彻底消灭了“上下文割裂”导致的幻觉。
3. **节省算力与存储**：短文档不用去建父子两套索引，节约了 KV 数据库的空间和处理时间。

### 工程落地的一个小坑
在设定阈值（`THRESHOLD`）时，一定要参考所选用的 Embedding 模型的 **Max Token Limit**。比如 OpenAI 的 `text-embedding-3-small` 支持 8191 个 Tokens，而很多开源模型（如早期的 BGE）只支持 512 个 Tokens。如果短文档的长度超过了 Embedding 模型的极限，它会在底层被强制截断，导致句尾的重要信息丢失。因此，阈值设定通常建议在 400~800 字符之间。
