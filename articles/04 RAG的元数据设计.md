### 一、 RAG 中的元数据 (Metadata)
最基础的 RAG 只关心文本本身，但在成熟的工业方案中，**任何一段被切分的文本块（Chunk），都必须挂载极其丰富的元数据**。

在向量数据库中，一条完整的记录通常长这样：

+ **Vector (密集向量)**：`[0.12, -0.45, 0.89...]` （用于计算语义相似度）
+ **Text (文本载荷)**：`"员工转正后可享受每年 5 天带薪年假。"`
+ **Metadata (元数据 - JSON 格式)**：
    - `doc_id`: `XYZ-998` (溯源用)
    - `author`: `HR_Dept` (权威度计算)
    - `create_time`: `2025-10-01` (时效性过滤)
    - `version`: `v2.1` (版本控制)
    - `security_level`: `L2_Internal` (权限控制 RBAC)
    - `category`: `Policy` (业务路由)

#### 企业内元数据的三大核心应用场景：
1. **标量预过滤 (Pre-filtering)：** 这是最常用的场景。用户提问：“最新的报销标准是什么？” 如果没有元数据，向量库会把过去 5 年的报销标准全搜出来。有了元数据，后端代码在查询向量库时，会直接拼上过滤条件：`相似度 > 0.8 AND category = 'Finance' AND create_time >= '2025-01-01'`。在计算极其耗时的向量距离之前，先把无关数据干掉，既精准又省算力。
2. **细粒度权限控制 (RBAC)：** 用户的请求打到后端，Java 拦截器解析 JWT Token 拿到该用户的 `department_id`，然后强制在底层检索语句中加上 `Metadata.department = user.department_id`，从物理层面杜绝越权访问。
3. **精准溯源与引用 (Citation)：** 大模型生成答案后，必须告诉用户“我是参考了哪份文件的哪一页”。这些信息全靠元数据带出来。

**Java 实现示例 (LangChain4j 中的 Metadata 过滤)：**

```java
import dev.langchain4j.store.embedding.filter.Filter;
import static dev.langchain4j.store.embedding.filter.MetadataFilterBuilder.*;

// 构建基于元数据的过滤条件：只搜索 2025 年之后，且密级为公开的文档
Filter filter = and(
    metadataKey("year").isGreaterThanOrEqualTo(2025),
    metadataKey("security_level").isEqualTo("PUBLIC")
);

// 在执行相似度检索时，强制带上这个 Filter
List<EmbeddingMatch<TextSegment>> results = 
embeddingStore.findRelevant(queryVector, 5, 0.8, filter);
```

### 二、 资料冲突与矛盾的工业级解决方案
当资料库里同时存在“老版本规定（年假 5 天）”和“新版本规定（年假 10 天）”，且都被检索出来塞给了大模型，大模型极大概率会“精神分裂”或者胡乱捏造。

在工程上，我们通常通过 **“检索前置拦截”** 和 **“生成时 Prompt 约束”** 两道防线来解决。

#### 方案 1：利用元数据进行“版本/时间线”压制 (最彻底的解法)
知识的冲突，90% 是因为**版本更新**或**时效性**引起的。

+ **具体做法**：在检索出 Top-K 个文档片段后，后端代码先不要急着喂给大模型，而是先按 Metadata 里的 `update_time` 或 `version` 进行一次**内存中的排序和去重**。
+ 如果发现两个文档块描述的是同一个 `doc_id` 或同一个核心实体，**直接丢弃旧版本**，只保留时间戳最新的那一份。大模型根本看不到冲突的数据，自然就不会晕。

#### 方案 2：权威度加权重排 
如果冲突的不是版本，而是来源（比如：百度百科说 A，公司内部技术白皮书说 B）。

+ **具体做法**：在做 Re-ranking（重排序）时，把 Metadata 里的 `source` 字段加入打分逻辑。
+ 内部权威系统（如官方技术文档、HR 盖章文件）的权重系数调高；外部抓取数据（如论坛帖子）的权重系数调低。确保喂给大模型的上下文里，权威信息排在最前面。

#### 方案 3：Prompt 策略 (最后一道防线)
如果前两步都没过滤干净，冲突的数据不可避免地进入了 Prompt，我们就必须在系统提示词中**显式地教大模型如何处理矛盾**。

我们需要在 Prompt 中加入严苛的**冲突处理法则**：

```plain
你是一个严谨的企业知识库助手。请基于以下【参考资料】回答用户问题。

【重要约束法则】：
1. 如果参考资料中存在针对同一问题的不同或矛盾说法，请优先采纳带有最新日期的资料。
2. 如果日期无法辨别，请不要自行猜测标准答案！你必须客观地向用户列出这种冲突。
3. 冲突回复模板：“关于您的问题，目前的资料存在不同描述。根据资料A显示为...；但根据资料B显示为...。建议您向相关部门进一步核实。”

【参考资料】：
{{context}}
```

在企业应用里，**“暴露系统知识库的混乱”远比“AI 自信地给出一个错误答案”要安全得多**。这种策略还能反向推动业务部门去清理那些历史遗留的冲突文档。

#### 方案 4：Agent 交叉验证
对于极其核心的决策问题，可以启动两个 AI Agent 进行“辩论”。

+ Agent 1 拿着资料 A 得出结论；Agent 2 拿着资料 B 得出不同结论。
+ 引入一个“法官 Agent（Judge）”，将用户的原始问题和两份冲突的资料同时给法官，让法官分析冲突的根源（比如一个是针对正式员工的，一个是针对实习生的），并最终生成一份包含严密逻辑推理的综合报告。
